## Phase 2 — Security-Fixes verifizieren (15 min, kritisch!)

**Ziel:** Die drei geschlossenen Security-Lücken sind wirklich zu.

**Setup:** Zwei Test-Accounts: User A (client), User B (client). Login-Tokens im Browser-DevTools / curl merken.

### 2.1 `profile.update` — Fremdprofile
Als User A eingeloggt:
```bash
curl -X POST https://app.functional-wiehl.de/api/trpc/profile.update \
  -H "Authorization: Bearer <USER_A_TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{"name":"hacked"}'
```
**Erwartung:** Updated User A (nicht User B).  
Alter Angriffsvektor `{"userId":"<B_ID>","name":"hacked"}` ist jetzt **nicht mehr möglich** — `userId` ist nicht im Schema.  
**Pass-Kriterium:** User B unverändert, User A hat jetzt name="hacked".

### 2.2 `auth.updatePassword` — Ohne Auth
Ohne Token:
```bash
curl -X POST https://app.functional-wiehl.de/api/trpc/auth.updatePassword \
  -H "Content-Type: application/json" \
  -d '{"currentPassword":"x","newPassword":"abcdef"}'
```
**Erwartung:** `UNAUTHORIZED` (401).  
**Failure:** `200 success` oder `USER_NOT_FOUND` → Fix nicht angekommen, Backend nicht neugestartet.

### 2.3 `gamification.sync` — Fremde XP
Als User A:
```bash
# Versuch, User B auf XP 9M zu setzen: alter Payload mit userId
curl -X POST https://app.functional-wiehl.de/api/trpc/gamification.sync \
  -H "Authorization: Bearer <USER_A_TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{"userId":"<USER_B_ID>","xp":9000000,"level":999,...}'
```
**Erwartung:** Input-Validation-Fehler (`userId` nicht im Schema) oder stilles Ignorieren — User B's XP unverändert.  
**Pass-Kriterium:** User B's `/api/trpc/gamification.get` zeigt unveränderte XP.

### 2.4 Credentials nicht mehr im Repo
```bash
git grep "LKW_Peter123" && echo "LEAK!" || echo "clean"
git grep "J5Kbw4LGHd79brs0MAfq9r4e63KSeQQkGmKQSqtFMzmQrVdk" && echo "LEAK!" || echo "clean"
```
**Erwartung:** Beide Male `clean`.
